### EMR-715: P0: /status page has hydration mismatch + duplicate React keys + dead footer buttons

* **Status:** `In Progress`
* **Priority:** P1

#### Description

Click-handler crawler (pass 8) caught three distinct bugs on `/status`. The page is in `src/app/status/status-view.tsx`.

Captured by `e2e/click-handlers.spec.ts`. Full output: `docs/audit/CLICK_HANDLERS_2026-05-17.md`.

## Bug 1 — server/client timestamp mismatch

```
Warning: Text content did not match.
  Server: "11:22:34 AM"
  Client: "11:22:37 AM"
  at StatusView (src/app/status/status-view.tsx:114:90)
```

Status page is rendering current time on the server then a different time on the client. Classic SSR/hydration anti-pattern — either:

* Move the timestamp to a client-only `useEffect`-initialized state, OR
* Pass the server-rendered time via props and only update client-side (with `suppressHydrationWarning`).

## Bug 2 — duplicate React keys

```
Warning: Encountered two children with the same key, `%s`.
Keys should be unique so that components maintain their identity across updates.
```

Fires multiple times per click. Find the `.map(... key={...})` in `status-view.tsx` (or its imports) where the key is repeating. Likely a list deriving keys from a non-unique field (e.g., status name without a stable id).

## Bug 3 — Hydration error

```
Warning: An error occurred during hydration. The server HTML was replaced with client content.
```

Cascade of bug 1 — once text mismatches, React bails and re-renders the whole subtree client-side, which is visible to users as a flicker.

## Bug 4 — Footer `Product+` and `Company+` buttons fail to click

```
locator.click: Timeout 5000ms exceeded.
locator: body > div > footer > div > div:nth-of-type(2) > div:nth-of-type(1) > button
```

Two footer accordion-style buttons in `SiteFooter` don't respond to Playwright clicks within 5s. Likely they're hidden on desktop (mobile-only accordions) and visible-but-blocked, OR they have a `pointer-events: none` parent. Verify visibility/interaction-state in dev tools.

## Repro

```bash
BASE_URL=http://localhost:3000 npx playwright test e2e/click-handlers.spec.ts -g 'crawl /status'
```

Also visible in any browser: open `/status`, watch console for the duplicate-key + hydration warnings, click the time-stamped element.

## Why P0

`/status` is the public uptime/status page — it should be the single most reliable page on the site. Hydration errors here flash a console full of red on every commercial visitor's first impression.

## Acceptance

* `/status` shows zero hydration warnings
* No duplicate React keys in console
* Footer `Product+` / `Company+` buttons either work or are removed
* `npx playwright test e2e/click-handlers.spec.ts -g 'crawl /status'` returns 0 medium-severity findings

### EMR-758: P2: dev-server stale-cache produces ghost 500s referencing files from other branches

* **Status:** `In Review`
* **Priority:** P3

#### Description

During the QA blitz the dev server returned **HTTP 500 on every public route** with:

```
ModuleBuildError: Module build failed (from ./node_modules/next/dist/build/webpack/loaders/next-swc-loader.js):
Error: Failed to read source code from /Users/scottwayman/ANTIGRAVITY/EMR/src/components/auth/IdleTimeoutGuard.tsx
Caused by: No such file or directory (os error 2)
```

…but `grep -r IdleTimeoutGuard src/` returned **zero** matches, and the `src/components/auth/` directory didn't exist on `feature/wave-9-sprint`. The reference came from PR #366 (HIPAA session auto-logoff, landed in `main`) which the running dev server had previously compiled against a different branch state.

`rm -rf .next/cache && kill PID && npm run dev` resolved the issue cleanly. Site immediately returned to 200 site-wide.

## Why P2

This is a **test-harness reliability issue**, not a product bug — but it's the kind of bug that produces false-positive 500 findings in any QA suite that runs against a long-lived dev server while the auto-agent is merging PRs in parallel. The marketplace search/filter spec (`e2e/marketplace-search-filter.spec.ts`) failed 8 of 9 tests during this episode purely because the dev server was serving error pages.

## Suggested fix

Add a `playwright.config.ts` global setup that:

1. Restarts the dev server (or clears `.next/cache`) before any test run, OR
2. Probes `/api/health` and aborts the run with a clear message if the response indicates a build error, OR
3. Uses `npm run build && npm run start` for E2E instead of `npm run dev` — production builds don't have the stale-cache failure mode.

Option 3 is the right long-term answer; option 2 is a fast defensive guard for the meantime.

[EMR-709](https://linear.app/emr-project/issue/EMR-709/qa-blitz-pre-commercial-release-polish-2026-05-17) parent. Surfaced by find-and-fix pass 10 (marketplace search/filter coverage).

### EMR-759: P3: /marketplace?category=&lt;unknown&gt; silently shows the full catalog instead of an empty/notfound state

* **Status:** `In Review`
* **Priority:** P4

#### Description

`src/app/marketplace/page.tsx:38-41`:

```typescript
if (categorySlug) {
  const cat = CATEGORIES.find((c) => c.slug === categorySlug);
  if (cat) products = products.filter((p) => p.categoryIds.includes(cat.id));
  // ← else: no filter applied, full catalog rendered
}
```

When a visitor lands on `/marketplace?category=anxiety-old` (e.g., a stale bookmark from before a slug rename, or a typo'd link from a campaign), the filter silently drops and the entire catalog renders — with the URL still claiming a category filter is active. The user has no way to know their bookmark is bad.

## Current behavior

```
$ curl -s -o /dev/null -w "%{http_code}\n" 'http://localhost:3000/marketplace?category=does-not-exist'
200
```

(Renders identical to `/marketplace` with no filter.)

## Proposed behavior

Pick one:

1. **404** — `notFound()` if the slug doesn't match. Strongest signal to the visitor (and to crawlers); matches how `/leafmart/category/[slug]/page.tsx` handles it now.
2. **Empty state with banner** — render the listing chrome but with "No products found in this category" + a "Browse all" link. Better for soft-rename cases.
3. **301 to /marketplace** — redirect to the unfiltered listing.

Option 2 is the most forgiving for legitimate users; option 1 is best for SEO hygiene.

## Why P3

Edge case. Not a launch blocker — bogus URLs are rare in practice — but it's the kind of papercut a paying customer might hit and silently blame the catalog for being wrong.

## Acceptance

* `/marketplace?category=does-not-exist` either 404s OR renders an empty-state banner.
* `e2e/marketplace-search-filter.spec.ts` `?category=nonexistent-slug` test flips its assertion to match.

Source: find-and-fix pass 10 — marketplace search/filter coverage. Related parent: [EMR-709](https://linear.app/emr-project/issue/EMR-709/qa-blitz-pre-commercial-release-polish-2026-05-17).

### EMR-719: E4 — Super Admin Hardening

* **Status:** `Backlog`
* **Priority:** P1

#### Description

## Why this epic ships first

Every other epic in Super Admin / Fleet Ops v1 puts richer surfaces in front of more eyes. None of them should land until the security floor is poured. Today the surface is gated (`requireSuperAdmin`, lazy-promotion from `SUPER_ADMIN_BOOTSTRAP_EMAILS`, last-admin guard, rate-limited mutations on 5 admin routes), but five real holes remain:

1. **No MFA enforcement on super_admin** — a compromised password is a compromised tenant fleet.
2. **No** `SUPER_ADMIN_BOOTSTRAP_EMAILS` **rotation workflow** — env var changes silently between deploys with no audit.
3. **No emergency revoke + session pinning** — Serializable revoke txn flips the role bit but in-flight sessions can keep working until cookie expiry.
4. **Controller rate-limit coverage is partial** — only 5 admin routes use `adminMutationLimiter`; the wider controller surface (\~15 routes from Onboarding Controller v1) has none.
5. **No per-actor mutation budget alarm** — a super_admin who suddenly issues 200 mutations/min is the loudest possible compromise signal and we ignore it.

## Acceptance gate for the epic

* A new super_admin cannot complete first sign-in without enrolling a second factor.
* Changing `SUPER_ADMIN_BOOTSTRAP_EMAILS` between two deploys emits an alert + audit row, even if nobody was actually promoted.
* An emergency revoke kills the target's live sessions inside one second, fleet-wide.
* Every controller mutation route — old and new — is behind a rate limiter and emits structured audit on every call.
* Any super_admin who exceeds N mutations/min triggers an alarm in our incident channel.

#### Description

# LeafBridge — Open-Source FHIR-Native Health Data Infrastructure

**Epic owner:** Scott (founder)
**Status:** Proposal → planning
**Date opened:** 2026-05-18

## 1\. Thesis

LeafBridge is an **open-source, FHIR-native clinical data lakehouse + AI orchestration layer** that lets clinics, specialty practices, and care teams unify patient data, govern access, and safely route that data into AI-powered clinical and administrative workflows.

Positioning (one line):

> **FHIR-native infrastructure for safe clinical AI.**

Legacy EHRs are systems of record. LeafBridge becomes the **system of intelligence and coordination** — a modern HIE paired with a governed agent operating system.

The moat is not the chatbot. The moat is the trust fabric (identity, consent, audit, provenance), the FHIR-native data model, the agent routing layer, and the specialty workflow integration.

LeafBridge is the open-source infrastructure project; **LeafJourney** is the commercial application layer on top.

## 2\. Strategic problems solved

1. Healthcare data is trapped in EHRs, labs, portals, PDFs, devices, and specialty systems.
2. AI cannot safely operate without structured, permissioned, auditable data access.
3. Clinical teams lack workflow-specific intelligence, especially in specialty medicine.
4. Existing HIEs move data but do not reason over it.
5. Most AI tools bolt onto the EHR rather than acting as a governed clinical operating system.

## 3\. Architecture — five layers

### Layer 1 — Ingestion & Interoperability

* FHIR-first ingestion (Patient, Encounter, Observation, Condition, MedicationRequest, DiagnosticReport, DocumentReference, CarePlan, Appointment, Coverage, Consent, Provenance, AuditEvent)
* HL7v2 / C-CDA / PDF / CSV adapters via OpenHIM + HAPI HL7v2
* Semantic normalization (SNOMED, LOINC, RxNorm, ICD-10, CPT, UCUM, NPI)
* Edge processing: PHI minimization, tokenization, redaction, validation, quarantine, "do not train" tags
* **Principle:** never overwrite source truth — store raw, parsed, normalized, and provenance separately.

### Layer 2 — Clinical Lakehouse

* Zones: Bronze (raw) → Silver (parsed) → Gold (FHIR) → Platinum (clinical marts) → Vector (AI retrieval) → Audit
* Stack: S3-compatible / MinIO, Apache Iceberg, Trino, Spark/Flink, Postgres, OpenSearch, pgvector/Qdrant, Kafka/Redpanda, Temporal, OpenTelemetry + Prometheus + Grafana
* Three persistence representations per resource: canonical FHIR JSON + relational projection + AI retrieval projection
* FHIR Bulk Data API for population-level export/import

### Layer 3 — Trust Layer (identity & governance)

* **Master Patient Index:** deterministic + probabilistic + referential matching, golden record, merge/split with audit
* **Consent & policy:** FHIR Consent resource, Open Policy Agent, purpose-of-use enforcement, sensitive-data segmentation
* **Agent identity:** every AI agent is a non-human workforce member with role, allowed tools, allowed patients, allowed data classes, autonomy tier (0–5), escalation rules, and full audit trail
* Stack: Keycloak (OIDC/SAML), OPA, Vault, SPIFFE/SPIRE, Kong/Envoy
* Zero-trust: every patient record, document, note, and AI action is a protected resource

### Layer 4 — AI Orchestration Engine

* Services: Agent Registry, Tool Registry, Context Broker, RAG Service, Policy Gatekeeper, Execution Engine, Human Review Queue, Evaluation Service, Audit Ledger
* FHIR Subscriptions trigger agent workflows (new lab, abnormal observation, missing prior auth, overdue follow-up, med rec gap)
* **RAG principles:** no naked prompts; every output cites patient-specific source data; minimum necessary context; structured data beats embeddings when available; embeddings support recall, FHIR supports truth; no clinical write-back without policy + human review
* Agent families: clinical (chart summarizer, lab interpreter, med rec, referral, prior auth, care gap, specialty protocol, patient education), administrative (intake, eligibility, scheduling, RCM, document, message triage), safety (PHI boundary, clinical critic, citation verifier, bias monitor, consent enforcer, escalation)
* **Agent output schema:** summary, recommendation, evidence, source references, confidence, risk level, required human action, write-back target, audit ID

### Layer 5 — Application & Execution

* Mission Control Dashboard, Patient 360, Agent Workbench, Care Coordination Routing
* Specialty Workflow Shells: Internal Medicine, Pain Management, Cannabis Medicine, Endocrinology/Diabetes, Behavioral Health adjacency
* Specialty differences handled via **workflow configuration DSL**, not hardcoded product forks

## 4\. MVP scope — "The Bridge" (v0.1)

**MVP goal:** A patient record can be ingested, normalized into FHIR, stored in the lakehouse, governed by consent, retrieved by an AI agent, summarized with citations, and routed into a human approval workflow.

### EMR-724: E6 — SaaS Billing + AI Model Brokering (v2)

* **Status:** `Backlog`
* **Priority:** P3

#### Description

## Why this epic exists — and why it's the biggest revenue lever

This is the v2 epic. It turns Leafjourney from a great clinical product into a SaaS + AI company with a real, defensible top-line revenue story. Two monetization layers stacked:

### Layer 1 — SaaS subscriptions of the practices themselves

Today there is no model for charging Leafjourney's customers (the practices) on a recurring basis. No per-practice plan, no seat enforcement, no MRR/ARR/churn. The HQ dashboard from E1 has an ARR placeholder tile that goes empty until this lands.

### Layer 2 — AI model brokering + token arbitrage

This is a **first-class monetization pillar**, not an afterthought. Today each agent calls the model provider directly. That works clinically but burns through any margin we'd otherwise capture. The plan:

* Route every per-practice LLM call through a central brokering layer (OpenRouter / direct Anthropic / direct OpenAI configurable per-tenant).
* Meter every call with per-model upstream cost, latency, output tokens, and which agent invoked it.
* Bundle token allowances into subscription plans. A practice on the $X tier gets Y tokens included.
* Calculate per-practice and fleet-wide gross margin: (retail token allowance × plan price) − (actual upstream cost × tokens consumed).
* Soft cap at 90% of plan tokens (alert) and hard cap at 110% (auto-throttle with override) so a runaway agent can't destroy plan margin overnight.

Done right, this becomes a recurring 30–60% gross margin line item on top of the SaaS subscription itself — the same arbitrage that powers the largest AI infrastructure businesses today.

## Acceptance gate for the epic

* A super-admin can see, for any practice: subscription tier, MRR, seat count, plan-included tokens, tokens consumed MTD, projected end-of-month overage, current month margin.
* A super-admin can see, fleet-wide: ARR, MRR, net new MRR, churned MRR, fleet token gross margin %, top 10 most expensive practices by upstream cost.
* A new provider seat cannot be added when over plan limit; the block emits a clear upgrade prompt.
* Token consumption above 90% of plan emits an alert; above 110% auto-throttles unless explicitly overridden by a super-admin.

## Design principles

* **One central brokering layer** — agents do not call providers directly. They call our broker. The broker logs and routes.
* **Cost is structured, not estimated** — we store actual upstream invoice line items, not back-of-envelope guesses.
* **Override-able** — auto-throttle exists, but a super-admin can always override for a specific practice (e.g. enterprise customer).

## Children (6)

Created as separate tickets under this epic. **Deferred to v2** — do not pull into the v1 sprint without explicit re-scoping.

* **Status:** `Backlog`
* **Priority:** P3

#### Description

## Goal

Every per-practice LLM call should flow through one central broker module that resolves which upstream provider/model to use for that practice's plan, records the call with token usage and upstream cost, and returns the response. This is the chokepoint that makes token metering, margin tracking, and per-practice cost guardrails possible — and it converts model spend from a fixed operating expense into a metered, marginable revenue line.

## Acceptance criteria

* Broker lives at `src/lib/ai/broker/index.ts` and exposes a typed `invoke({ practiceId, agentBucket, model, messages, temperature?, tools?, ... })` returning a typed response that mirrors the existing per-route shapes.
* New `PracticeAiConfig` Prisma model: `organizationId` (unique), `defaultUpstream` (enum: `openrouter` | `anthropic` | `openai`), `perAgentOverrides` (JSONB keyed by agent bucket), `throttled` (boolean, default false), timestamps. Default row seeded for every org.
* Broker resolves upstream as: per-agent override -> org default -> registry fallback. Throttled orgs short-circuit to a structured 429 (consumed by Child 6, but the broker must already honor the flag here).
* Every existing direct LLM call in `src/app/api/agents/**/route.ts` (and any other agent route that today uses `OPENROUTER_API_KEY` directly) is migrated to call the broker. Maintain a migration checklist in this issue description, ticking each route as it lands; no route ships partially migrated.
* New `LlmUsage` Prisma model written per call: `practiceId`, `agentBucket`, `model`, `upstream`, `promptTokens`, `completionTokens`, `upstreamCostCents`, `latencyMs`, `requestId`, `createdAt`. Cost is computed from a typed per-model rate card colocated with the tier registry.
* Failure modes (upstream timeout, rate limit, malformed response) still write a `LlmUsage` row with the failure annotated; partial token counts are recorded when available.
* Existing rate limiting via `agentInvocationLimiter` and auth via `requireApiAuth` remain the outermost gate — broker sits inside those.
* Unit tests cover upstream resolution, override precedence, throttle short-circuit, and usage-row shape on both success and failure.

## Notes

### EMR-756: Per-practice cost guardrails (soft + hard caps)

* **Status:** `Backlog`
* **Priority:** P3

#### Description

## Goal

Protect both the customer and our margin: a practice approaching its plan's included token allowance should get a warning; a practice well past it should auto-throttle, unless a super-admin has explicitly authorized overage (the enterprise case with negotiated terms). Without this, a single runaway agent loop on a starter plan could erase a month of margin.

## Acceptance criteria

* Soft cap at 90 percent of `PracticeSubscription.includedMonthlyTokens` (computed against MTD `LlmUsage`) emits an `approaching_token_cap` anomaly via the E2 anomaly framework at severity `warning`. Anomaly payload includes practice ID, current usage, included allowance, and projected month-end usage. Re-emission is idempotent within a single billing period.
* Hard cap at 110 percent of included tokens sets `PracticeAiConfig.throttled = true`. The broker (Child 4) already honors this flag and short-circuits new calls with HTTP 429 body `{ code: "token_cap_exceeded", currentTokens, includedTokens, overageTokens }`.
* Cap evaluation runs on a short interval (cron or post-invocation hook) — must catch a practice within one to two minutes of crossing a threshold; document the chosen mechanism in the ticket on land.
* Super-admin UI at `/admin/practices/[id]/costs` exposes an "Allow overage" toggle per practice. Toggling on clears `throttled` and records the override; toggling off restores enforcement.
* Override actions emit a `super_admin.token_cap_override` audit row including actor user ID, previous and new state, and a free-text reason field that is required at the UI layer.
* When a practice is in the throttled state, the per-practice Costs tab (Child 5) shows a banner: "AI calls are currently throttled — plan token allowance exceeded" with a link to the override toggle.
* Period rollover: at the start of each new billing period (Stripe-driven, off `currentPeriodStart`), `throttled` is automatically cleared unless an explicit "persistent override" is set on the override row.

## Notes

v2 scope; deferred from v1. Depends on Child 4 (broker + `PracticeAiConfig.throttled`) and on E2 Child 2 anomaly framework ([EMR-734](https://linear.app/emr-project/issue/EMR-734/anomaly-detector-framework-severity-ttl)). Together with Child 5 this closes the loop: meter -> visualize -> protect. The reason field on the override is not cosmetic — it is how Finance will defend negotiated overage decisions during audits later.

### EMR-778: LeafBridge Phase 0.8: Extend Specialty Template + Practice Config schemas (agents / routing / writeback)

* **Status:** `Backlog`
* **Priority:** P3

#### Description

## Status

**Rescoped 2026-05-19** after compatibility audit (see comment thread). The original "draft a new DSL" framing duplicated work already underway in the Practice Onboarding Controller v1 project ([EMR-407](https://linear.app/emr-project/issue/EMR-407/epic-1-practice-onboarding-controller-wizard-state-machine)..479). This issue now extends those existing schemas with LeafBridge-specific sections instead of introducing a parallel config format.

Parent: [EMR-762](https://linear.app/emr-project/issue/EMR-762/leafbridge-fhir-native-clinical-data-lakehouse-ai-orchestration) (LeafBridge)
Phase: 0 (Foundation Design)
Blocked by: [EMR-408](https://linear.app/emr-project/issue/EMR-408/epic-2-specialty-template-registry) (Specialty Template Registry), [EMR-409](https://linear.app/emr-project/issue/EMR-409/epic-3-practice-configuration-object) (Practice Configuration Object), [EMR-410](https://linear.app/emr-project/issue/EMR-410/epic-4-modality-control-layer) (Modality Control Layer)

## Goal

Add three new sections to the existing **Specialty Template manifest** ([EMR-408](https://linear.app/emr-project/issue/EMR-408/epic-2-specialty-template-registry)) and **Practice Configuration Object** ([EMR-409](https://linear.app/emr-project/issue/EMR-409/epic-3-practice-configuration-object)) schemas so that LeafBridge agents, FHIR-event routing, and write-back policy are first-class config — not a parallel DSL.

## What we are NOT doing

* Not introducing a new DSL file format
* Not introducing a new per-practice config store
* Not introducing a parallel modality / enable-disable mechanism
* Not introducing a new manifest loader

LeafBridge reads `PracticeConfiguration` via the typed `getPracticeConfig(practiceId)` client defined in [EMR-409](https://linear.app/emr-project/issue/EMR-409/epic-3-practice-configuration-object). Single source of truth.

## What we ARE adding

### 1\. `agents[]` on the Specialty Template manifest ([EMR-408](https://linear.app/emr-project/issue/EMR-408/epic-2-specialty-template-registry) extension)

```yaml
agents:
  - id: previsit_summary
    autonomy_tier: 2          # 0–5 per EMR-768
    modality: ~               # ~ = always enabled; or "cannabis-medicine" etc.
    allowed_data_classes:
      - conditions
      - medications
      - observations
      - documents
    allowed_tools:
      - fhir.read
      - rag.query
    purpose_of_use: treatment
    requires_human_review: true
    escalation:
      on_risk_above: moderate
      route_to: clinical_triage_queue
```

* Agents self-register the same way [EMR-411](https://linear.app/emr-project/issue/EMR-411/epic-5-shell-rendering-engine) modules do (descriptor + `visibleWhen(config)` predicate)
* Modality dependency goes through `isModalityEnabled(practiceId, agent.modality)` — same gate as UI
* Default-null on existing rows; no breaking migration

### 2\. `agent_enable_overrides` on the Practice Configuration Object ([EMR-409](https://linear.app/emr-project/issue/EMR-409/epic-3-practice-configuration-object) extension)

```yaml
agent_enable_overrides:
  previsit_summary: enabled
  opioid_risk_review: disabled
```

Practice-level overrides on the template default. Audited via existing config versioning ([EMR-416](https://linear.app/emr-project/issue/EMR-416/epic-10-configuration-versioning-and-audit)).

### 3\. `clinical_routing_rules[]` on the Specialty Template + Practice Configuration

```yaml
clinical_routing_rules:
  - name: high_pain_score
    when:
      resource: Observation
      code: pain_score
      predicate: { value_greater_than: 8 }
    then:
      route_to: clinical_triage_queue
      priority: high
      trigger_agent: opioid_risk_review   # optional
```

Triggered by FHIR Subscriptions from Module 6 ([EMR-768](https://linear.app/emr-project/issue/EMR-768/leafbridge-module-6-agent-orchestrator)). Lives on the template as default and overridable on `PracticeConfiguration`, mirroring the `default_workflows[] → workflow_template_ids[]` pattern from [EMR-408](https://linear.app/emr-project/issue/EMR-408/epic-2-specialty-template-registry)/409.

### 4\. `writeback_policy` on the Specialty Template + Practice Configuration

```yaml
writeback_policy:
  allowed_resources:
    - CarePlan
    - ServiceRequest
    - DocumentReference
  requires_approval: true
  max_autonomy_tier: 3
```

Consumed by the Consent & Policy Gateway ([EMR-767](https://linear.app/emr-project/issue/EMR-767/leafbridge-module-5-consent-and-policy-gateway)), not by the controller. The controller just stores it.
