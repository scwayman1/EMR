# Encryption & Compliance Framework (EMR-084)

This document is the auditable companion to
`src/lib/security/encryption-framework.ts`. It exists so a HIPAA, SOC 2, or
HITRUST auditor (or a customer's privacy officer) can answer the question
"what algorithms does Leafjourney EMR use to protect PHI, and where is that
implemented in code?" without grepping the repo.

## 1. Scope

- **In scope:** PHI fields stored at rest in the application database,
  document uploads in object storage, sensitive AuditLog metadata,
  patient-portal session tokens, password hashes, and de-identified
  research export keys.
- **Out of scope:** Network-level encryption (handled by the platform load
  balancer / HSTS / TLS 1.2+), backup encryption (handled by Postgres
  managed backups), and KMS root-key custody (provider-managed).

## 2. Algorithms

| Purpose                     | Algorithm           | Parameters                                |
| --------------------------- | ------------------- | ----------------------------------------- |
| Symmetric encryption (PHI)  | AES-256-GCM         | 96-bit IV, 128-bit auth tag               |
| Key derivation              | HKDF-SHA-256        | Empty salt, purpose-string info parameter |
| Password hashing            | scrypt              | N=16384, r=8, p=1, keylen=64              |
| Hash digests                | SHA-256             | —                                         |
| Integrity / tamper evidence | HMAC-SHA-256        | Subkey derived from KEK via HKDF          |
| Constant-time comparison    | `timingSafeEqual`   | —                                         |

All choices conform to **NIST SP 800-175B Rev. 1** and HHS Breach Safe Harbor
guidance.

## 3. Envelope format

PHI fields are stored as a single string in this format:

```
v1:<iv-b64u>:<tag-b64u>:<wrappedDek-b64u>:<ct-b64u>
```

- `v1` — envelope version. Bumping the version is forward-compatible: the
  old decoder remains alongside the new one until backfill completes.
- `iv` — 12 random bytes used by AES-GCM.
- `tag` — 16-byte AES-GCM authentication tag.
- `wrappedDek` — `iv ‖ tag ‖ ciphertext` for the per-record DEK, encrypted
  under a purpose-specific subkey derived from the master KEK.
- `ct` — actual encrypted plaintext.

Per-record DEKs mean a single-record key compromise doesn't leak any other
record. Only the single KEK in KMS is shared.

## 4. Key custody

The key-encryption key (KEK) lives in the cloud KMS in production. The
application receives it as 32 random bytes via the `EMR_PHI_KEK` env var,
delivered by the platform's secret-manager sidecar.

- **Rotation cadence:** Annually, or immediately on personnel change with
  KMS access. Old envelopes are re-encrypted by a background job; old KEKs
  are retained for 90 days for decryption-only.
- **Local development:** `EMR_PHI_KEK` is set to a per-developer key
  generated by `scripts/dev-encryption-key.ts`. Production keys are never
  copied to dev environments.

## 5. Audit & monitoring

Audit controls (HIPAA §164.312(b)) are enforced by:

- The `AuditLog` Prisma model — every write to PHI flows through
  `recordChartAccess()` or `writeAgentAudit()`.
- `complianceAuditAgent` (`src/lib/agents/compliance-audit-agent.ts`)
  scans the AuditLog daily for compliance signals: off-hours access,
  high-volume single-actor access, sensitive-record break-glass, auth
  failure bursts, bulk exports, and uncoupled PHI disclosures.

Findings are reviewed weekly by the privacy officer through
`/ops/audit-trail` and exportable to PDF (EMR-064).

## 6. Sensitive record overlay

Mental-health, substance-use, and other 42 CFR Part 2 / state-protected
records pass through `src/lib/clinical/mental-health-access.ts`, which
classifies the record's sensitivity and decides whether the viewer must
break-glass. The break-glass attestation requires a typed e-signature, an
acknowledged purpose, and an acknowledged redisclosure-rules check; the
event is written to AuditLog as `phi.sensitive.break_glass`.

## 7. Compliance control mapping

The authoritative mapping lives in `COMPLIANCE_CONTROLS` in
`encryption-framework.ts`. A summary:

| Control                     | Citation                                      | Implementation                                                        |
| --------------------------- | --------------------------------------------- | --------------------------------------------------------------------- |
| Encryption at rest          | HIPAA §164.312(a)(2)(iv)                      | `encryptString` / `decryptString`                                     |
| Encryption in transit       | HIPAA §164.312(e)(2)(ii)                      | TLS 1.2+ at platform load balancer                                    |
| Audit controls              | HIPAA §164.312(b)                             | `AuditLog` + `complianceAuditAgent`                                   |
| Breach safe harbor          | HHS Breach Safe Harbor                        | AES-256-GCM envelope                                                  |
| Logical access controls     | SOC 2 CC6.1                                   | RBAC + mental-health access wall                                      |
| Anomaly monitoring          | SOC 2 CC7.2                                   | `complianceAuditAgent`                                                |
| Cryptographic algorithms    | NIST SP 800-175B Rev. 1                       | Constants in `encryption-framework.ts`                                |
